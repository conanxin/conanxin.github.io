# 内容治理政策

> 《后来礼物 / Aftergift》内容治理核心文档
> 版本：1.0 | 状态：原型阶段 · 待 Phase 2 完善

---

## 核心理念

**"我们处理的是礼物，不是审判一个人。"**

用户可以讲述关系和感受，但不应暴露他人身份或将平台用于报复。礼物的下一站比过去的故事更重要。

---

## 一、平台价值观

1. **聚焦物品**：故事的主角是礼物，不是关系中的对方
2. **聚焦感受**：允许表达失落、遗憾、释然，但不鼓励控诉
3. **匿名优先**：不强制披露任何关系类型，匿名发布是默认选项
4. **温柔告别**：礼物值得体面的流转，故事也是
5. **零容忍报复**：曝光、羞辱、网暴类内容不被允许

---

## 二、鼓励内容

以下内容是平台鼓励的：

- ✅ 用「TA」「那个人」「曾经的某个人」等代称指代关系对象
- ✅ 聚焦礼物本身（它是什么、什么时候收到、现在怎样了）
- ✅ 描述自己的感受（遗憾、释怀、感谢、放下）
- ✅ 说明为什么想让礼物离开，以及希望它去往哪里
- ✅ 真实讲述，不过度戏剧化，不煽情渲染
- ✅ 让故事成为自我梳理和疗愈的一部分
- ✅ 选择匿名发布，不暴露任何关系细节
- ✅ 用一件有故事的礼物交换另一件有故事的礼物

---

## 三、不允许内容

以下内容明确禁止，发布即违规：

### 3.1 身份信息

- ❌ 真实姓名（全名或姓氏+名）
- ❌ 手机号码（11 位手机号，任何格式）
- ❌ 微信号 / 微信公众号
- ❌ QQ 号
- ❌ 邮箱地址
- ❌ 详细地址（小区、楼栋、单元、门牌号）
- ❌ 公司 / 学校 / 医院 / 大厦 / 广场等具体场所 + 地名组合
- ❌ 社交平台账号（微博、抖音、小红书、Instagram、Telegram 等）

### 3.2 攻击性内容

- ❌ 侮辱、羞辱、诽谤性语言
- ❌ 报复、曝光、挂人、人肉搜索
- ❌ 诅咒、生日快乐变成诅咒（如"祝你孤独终老"）
- ❌ 将他人身份作为卖点或噱头（"前任的礼物，正主在 XX 公司"）
- ❌ 虚假信息或诈骗意图

### 3.3 传播风险内容

- ❌ 暗示或明确引导他人攻击特定个人
- ❌ 发布他人照片、截图等影像资料
- ❌ 发布可间接识别他人身份的组合信息

## 四、当前原型静态规则（Phase 1D）

### 4.1 身份信息风险检测

| 规则 | 匹配示例 |
|------|---------|
| 手机号 | `13812345678`、`1 381 234 5678`、`138-1234-5678` |
| 微信号 | `微信号：wxabc`、`wechat`、`加我微信` |
| QQ号 | `QQ：123456789`、`qq 987654` |
| 邮箱 | `user@example.com` |
| 身份证格式 | `110101 19900101 1234` |
| 地址关键词 | 小区、楼栋、单元、门牌、住所等 |
| 社交账号 | 微博账号、抖音号、小红书号、Instagram 等 |

### 4.2 报复/攻击性表达检测

| 规则 | 匹配示例 |
|------|---------|
| 控诉词汇 | 渣男、渣女、报复、曝光、挂人、人肉、骗子、毁掉 |
| 诅咒词汇 | 去死、该死、恨死、恶心死了、想打、想杀 |
| 侮辱词汇 | 贱、不要脸、无耻、下头男/女 |

### 4.3 可识别关系对象检测

| 规则 | 匹配示例 |
|------|---------|
| "XXX 叫 XXX" | `他叫张明`、`她叫李华` |
| "姓名：XXX" | `姓名：王五`、`Name: LiSi` |
| "全名 XXX" | `全名：陈六` |

### 4.4 故事完整度检查

| 检测项 | 触发条件 | 提示文案 |
|--------|---------|---------|
| 字数不足 | < 50 字 | 再写一点点，别人会更理解这件礼物为什么重要 |
| 缺少来历 | 无来/得到/收到/赠/送/买等词 | 可以补充一下礼物最初是怎么来到你手上的 |
| 缺少意义 | 无重要/喜欢/陪伴/记得/想起等词 | 如果能再写一句它曾经对你意味着什么，这个故事会更完整 |
| 缺少告别理由 | 无现在/离开/告别/不再/结束等词 | 可以再写一句为什么现在想让这件礼物离开 |
| 缺少下一站 | 无希望/愿/以后/下一位等词 | 如果你愿意，可以再补一句希望这件礼物去往怎样的下一站 |

### 4.5 匿名化建议（9 条）

| 类型 | 建议改法 |
|------|---------|
| 真实姓名 | 那个人 / TA / 我曾在乎的人 |
| 手机号码 | 后来我们不再联系了 |
| 微信号 | 我们后来失去了联系 |
| QQ号 | 我们后来失去了联系 |
| 邮箱 | 我们后来失去了联系 |
| 报复性词汇 | 这段关系让我失去了信任 / 让我感到受伤 |
| 诅咒表达 | 把感受写出来，但不要用伤害性的语言 |
| 可识别地点 | 那个人工作的地方 / 我们曾住在同一座城市 |
| 住址信息 | 我们曾经住得很近 / 后来搬到了不同的地方 |

## 五、未来真实审核层建议

当前原型规则为纯前端正则，无法拦截高级规避手段。Phase 2+ 建议以下架构：

### 5.1 服务端审核

- 所有故事在发布前经过服务端 API 审核
- 拒绝不符合规范的故事，返回修改建议
- 日志记录所有发布内容（合规留存）

### 5.2 AI 文本审核

推荐接入：
- **OpenAI Moderation API**：通用内容安全检测（免费，调用简单）
- **百度文本审核**：中文内容场景优化（付费，有敏感词库）
- **阿里云内容安全**：多语言支持（付费）

建议流程：
1. 用户提交 → 服务端接收
2. 调用 AI 审核 API（2-3 秒延迟）
3. 风险等级：SAFE / REVIEW / BLOCK
4. BLOCK 直接拒绝；REVIEW 进入人工复审队列；SAFE 发布

### 5.3 人工复审队列

- 高风险内容（涉及真实身份曝光、报复）进入人工队列
- 配备标注工具，标注人员可在 Web 后台查看
- 24 小时内处理
- 支持通过 Telegram Bot 通知管理员

### 5.4 风险分级

| 等级 | 定义 | 处理方式 |
|------|------|---------|
| P0 | 直接暴露真实身份（姓名+电话+地址） | 拒绝发布，要求修改 |
| P1 | 报复性内容或高度可识别信息 | 标记，进入人工复审 |
| P2 | 中等风险（间接可识别、小范围攻击） | 通过但标记，发布者收到温和提示 |
| P3 | 低风险（文字质量不足、字数过少） | 通过，通过 AI 建议引导用户补充 |

### 5.5 举报机制

- Modal 详情页有「举报隐私问题」按钮
- 用户填写举报原因（可选），提交
- 后端记录举报，触发人工复审
- 举报处理结果通过 Telegram Bot 通知管理员

### 5.6 申诉机制

- 如果用户的故事被错误拦截，可以提交申诉
- 管理员在 Web 后台审核申诉，48 小时内回复
- 申诉通过后故事自动发布
