Methodology

How PDFs are processed in this project

Pipeline Overview

This project uses two separate routes depending on the nature of the input PDF:

RouteInputOutput
Digital PDFDocuments with embedded text layersHTML / Markdown / Text
Scanned PDFImage-only documents requiring OCRImage-first HTML + OCR text

Digital PDF Pipeline

Digital PDFs are the most straightforward. The text is already present in the document, so extraction focuses on preserving structure — headings, paragraphs, lists, tables.

  • Text extraction via pdftotext (layout mode) and PyMuPDF
  • Table detection via pdfplumber
  • Image extraction via PyMuPDF
  • Markdown conversion with structure preservation
  • Single-file HTML generation with embedded images

This route produces high-quality, reliable output.

Scanned PDF Pipeline

Scanned PDFs require OCR because the pages are images, not text.

  • OCR engine: Tesseract + OCRmyPDF
  • Image rendering: pdftoppm for page images
  • Output: image-first HTML (pages displayed as images, OCR text available for search/copy)
  • Font support: Noto CJK fonts installed for CJK rendering

OCR text is treated as an auxiliary resource — useful for searching and copying, but not the primary presentation layer.

Current Limitations

  • Markdown alone is not suitable for fully reproducing the layout of scanned book pages
  • Font optimization improves rendering quality, not OCR recognition accuracy
  • For older scanned Chinese documents, Tesseract's character recognition may produce spacing and accuracy issues
  • Scanned document processing is still under active refinement

Publishing Principles

  • Digital outputs first — reliable quality, ready for public use
  • Scanned outputs are more cautious — reviewed internally before public release
  • No internal paths leaked — published content avoids exposing local file paths or work-in-progress details
  • Single-file HTML — all images embedded as data URIs, no external dependencies
