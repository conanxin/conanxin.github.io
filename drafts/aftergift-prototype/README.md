# 后来礼物 Aftergift — 原型

> 故事型礼物流转平台。把关系结束后难以处理的礼物，变成可以被讲述、流转、出售、交换或捐出的故事物。

## 项目简介

这是一个静态单页 Web App 原型，展示了「后来礼物」的核心交互流程和视觉风格。产品定位为「故事型礼物流转平台」——用户处理的不是单纯商品，而是关系结束或变化后的礼物，每件礼物背后都要写下一个故事。

**核心原则：** 温柔、克制、干净、有一点伤感。不鼓励报复、羞辱、猎奇或网暴。

## Phase 2D：匿名身份 + 管理员审核 UI（可选）

Phase 2D 在 Phase 2C API 联调基础上，增加了匿名身份认证和管理员审核 UI。

### 1. 匿名身份（Dev Auth Panel）

在 API 模式下，右下角会显示 Dev Auth Panel：

```
http://localhost:8080/?api
```

点击「创建匿名身份」按钮 → 调用 `POST /api/auth/anonymous` → 获得 token 并存入 `localStorage['aftergift_token']`。
之后所有发布/收藏/举报请求自动附加 `Authorization: Bearer {token}`。

### 2. 管理员审核 UI

访问审核面板：

```
http://localhost:8080/?api&admin=1
```

输入 admin token（开发环境：`dev-admin-aftergift-001`），加载审核队列，可对礼物执行 approve / needs_edit / reject 操作。

## Phase 2C：本地前后端联调（可选）

原型默认以 **static 模式**运行（读取 `data/gifts.json`），不依赖后端。

如需启用本地 API 联调：

### 1. 启动后端

```bash
cd ~/projects/aftergift-backend-mvp/backend
python scripts/init_db.py          # 首次需初始化数据库
. .venv/bin/activate               # 激活虚拟环境
uvicorn app.main:app --host 127.0.0.1 --port 8091
```

### 2. 启动前端

```bash
cd ~/projects/aftergift-prototype
python3 -m http.server 8080
```

### 3. 进入联调模式

在浏览器访问：

```
http://127.0.0.1:8080/?api
```

页面底部应显示：`当前：本地 FastAPI 联调模式`

### 4. 模式说明

| 模式 | 触发方式 | 数据来源 | GitHub Pages 兼容 |
|------|---------|---------|----------------|
| **static**（默认） | 无参数 | `data/gifts.json` | ✅ 完全兼容 |
| **api** | `?api` | FastAPI `127.0.0.1:8091` | ❌ 需本地后端 |

**详细文档：** 请阅读 [`docs/API_INTEGRATION.md`](docs/API_INTEGRATION.md)

## 隐私与伦理边界

原型层面已内置以下安全设计：

- 表单收集关系类型、情绪标签等有限信息，不收集姓名、地址、手机号
- **故事预检规则引擎**：前端自动检测手机号、微信号、QQ号、邮箱、身份证格式数字、详细地址关键词、报复性表达、可识别姓名模式，输出温柔提示
- **发布前确认 Checkbox**：3 项必选确认，强制用户自我审核内容
- **故事预览**：发布前可预览他人将看到的实际样式
- **故事模板引导**：提供 4 个写作提示，引导用户聚焦礼物和自己的感受，而非描述对方
- Modal 详情页包含安全提示文案
- 禁止曝光他人真实身份信息（姓名 / 照片 / 手机 / 地址 / 公司 / 社交账号）
- 禁止发布侮辱、报复、控诉式内容
- 鼓励匿名化叙述，聚焦物品和自己的感受
- 所有示例数据均为虚构

## 故事预检规则说明（Phase 1D — AI 编辑建议引擎）

预检引擎包含两大模块：**风险检测**和**匿名化建议**（纯前端静态规则，不上传任何数据）。

**A. 风险检测（三级输出）**

*身份信息风险*
- 手机号（1开头的11位数字组合，可含空格/短横线）
- 微信号、WeChat、wx/WX、加我微信、微信号等
- QQ号（5位以上数字）
- 邮箱地址
- 身份证格式数字
- 详细地址关键词（小区、楼栋、单元、门牌等）
- 社交平台账号关键词（微博、抖音、小红书、Instagram、Twitter、Telegram等）

*报复/攻击性表达风险*
- 渣男、渣女、报复、曝光、挂人、人肉搜索、人肉、骗子、毁掉等控诉类词汇
- 去死、该死、恨死、恶心死了、想打/杀/弄死等诅咒类表达
- 贱、不要脸、无耻、下头男/女等攻击性词汇

*可识别关系对象风险*
- "XXX 叫 XXX" / "TA 叫 XXX" 等姓名暴露格式
- "姓名：XXX" 格式
- "全名 XXX" 格式

**B. 匿名化建议（9 条规则）**

每条建议包含：原表达类型 / 为什么有风险 / 建议改法（温和替代表达）

- 真实姓名暴露 → 用"那个人/TA/我曾在乎的人"替代
- 手机号码 → "后来我们不再联系了"
- 微信号 → "我们后来失去了联系"
- QQ号 → "我们后来失去了联系"
- 邮箱地址 → "我们后来失去了联系"
- 报复性词汇（渣男/渣女等）→ "这段关系让我失去了信任 / 让我感到受伤"
- 诅咒类表达 → "把感受写出来，但不要用伤害性的语言"
- 可识别地点（公司/学校+具体地名组合）→ "那个人工作的地方 / 我们曾住在同一座城市"
- 住址信息（小区+楼栋等）→ "我们曾经住得很近 / 后来搬到了不同的地方"

**C. 故事完整度检查（5 项）**

- 字数 < 50 字：提示"再写一点点，别人会更理解这件礼物为什么重要"
- 缺少来历：提示"可以补充一下礼物最初是怎么来到你手上的"
- 缺少意义：提示"如果能再写一句它曾经对你意味着什么，这个故事会更完整"
- 缺少告别理由：提示"可以再写一句为什么现在想让这件礼物离开"
- 缺少下一站：提示"如果你愿意，可以再补一句希望这件礼物去往怎样的下一站"

## 后续可扩展方向

1. **真实后端**：用户系统 + 数据库持久化
2. **真实交易**：担保支付 + 物流追踪 + 交换撮合
3. **社区功能**：收藏故事 + 温和评论 + 匿名私信
4. **线下版**：城市礼物交换会 + 关系旧物展
5. **移动 App**：React Native 或 Flutter 版本
6. **内容审核层**：AI 辅助识别违规内容（含后端规则引擎）
7. **故事模板编辑器**：结构化引导（如：填空式模板、心情曲线记录）
8. **匿名私信系统**：不暴露真实身份的读者与发布者通信机制

---

*这是一个静态原型，所有故事均为虚构。平台不处理真实交易，不存储用户敏感信息。*
